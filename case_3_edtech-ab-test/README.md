# A/B‑тест персонализированной карусели курсов (EdTech)

## Описание проекта

Проект посвящён дизайну и оценке A/B-теста персонализированной карусели курсов
на EdTech-платформе.

Цель — проверить, увеличивает ли персонализация карусели «Вам понравится»:
- вовлечённость пользователей,
- конверсию в запись на курс,
- монетизацию.

Ключевая особенность проекта — **обоснованное решение не запускать A/B-тест**
на этапе дизайна из-за несоответствия данных требованиям эксперимента.

---

## Контекст и техническое задание

По техническому заданию требовалось:

- провести A/B-тест:
  - **Control** — стандартная (неперсонализированная) карусель;
  - **Test** — персонализированная карусель «Вам понравится»;
- проверить рост следующих метрик:
  - **Primary**: CR_enroll — конверсия в запись после клика по карусели;
  - **Secondary**: CTR, AOV, ARPU;
- ограничения эксперимента:
  - не более **20% трафика**,
  - длительность **7 дней**,
  - минимальный детектируемый эффект (MDE) ≈ **2.5%**.

---

## Данные

Для анализа использовался event-log файл:

**`edtech_events.csv`**

Формат данных:
- `dt` — дата события  
- `client_id` — идентификатор пользователя  
- `course_id` — идентификатор курса  
- `event_type` — тип события:
  - `view_carousel`
  - `click_course`
  - `enroll`
  - `pay`
- `price` — сумма оплаты (для события `pay`)

Согласно ТЗ, данные должны позволять собрать воронку:

view_carousel → click_course → enroll → pay

## Часть 1. Exploratory Data Analysis

В ноутбуке `1_exploratory_analysis.ipynb` выполнено:

- проверка качества и структуры данных;
- анализ пользовательской активности;
- формальная сборка воронки событий на уровне пользователя и по дням;
- проверка логической последовательности шагов воронки;
- оценка пригодности данных для расчёта ratio-метрик.

### Ключевые наблюдения:
- крайне низкая активность пользователей (≈ 1 событие на пользователя);
- наличие альтернативных путей:
  - `enroll` без `click_course`,
  - `pay` без `enroll`;
- отсутствие оплат среди пользователей,
  взаимодействовавших с каруселью;
- нулевая вариативность revenue-метрик.

---

## Часть 2. Дизайн эксперимента

В ноутбуке `2_experiment_design.ipynb`:

- зафиксированы метрики и критерии успеха согласно ТЗ;
- показано, что:
  - CR_enroll не интерпретируем из-за нарушения предпосылок воронки;
  - AOV и ARPU не имеют дисперсии;
- формально обоснована невозможность:
  - расчёта MDE ≈ 2.5%;
  - запуска A/B-теста в рамках 20% трафика и 7 дней.

---

## Ключевой вывод проекта

Несмотря на формальное соответствие структуры данных требованиям ТЗ,
исторические логи **не удовлетворяют статистическим и продуктовым предпосылкам**
для проведения корректного A/B-теста персонализированной карусели.

**Решение не запускать эксперимент на этапе дизайна
является корректным аналитическим выводом**, позволяющим избежать
ложных результатов и неверных продуктовых решений.
